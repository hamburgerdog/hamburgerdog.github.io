# Chinese Documentation Guide

这份文档是项目内的中文技术文档写作规范，来源于本地 `chinese-documentation` skill 的项目化摘录。新增、编辑、润色博客文章时，先读 `writing-guide.md`，再用本文做排版和术语检查。

## 核心原则

- 排版服务于阅读体验，规范服务于一致性，内容服务于读者。
- 技术文章先降低信息差，再追求表达风格。
- 不要有「机翻味」：避免长定语、被动语态和英文句式直译。

## 空格

- 中文与英文之间加空格：`使用 Git 进行版本管理`。
- 中文与数字之间加空格：`包含 3 个新功能`。
- 数字与单位之间加空格：`5 MB`、`200 ms`。
- 百分比、温度不加空格：`95%`、`32°C`。
- 链接前后加空格：`请参考 [官方文档](https://example.com) 获取更多信息。`

## 标点

- 中文语境使用全角标点：`注意：该接口需要鉴权，请先获取 Token。`
- 全角标点与英文、数字之间不加空格。
- 中文说明优先使用直角引号：`「确定」按钮`。
- 括号按语境选择：
  - 中文补充说明用全角括号：`请运行安装命令（详见下方说明）。`
  - 括号内是英文、版本号、参数时用半角括号：`Spring Boot (v3.2.0)`。

## 术语

保留英文的情况：

- 专有名词：React、Redis、MySQL、Codex、Astro。
- 行业通用缩写：API、SDK、CLI、ORM、CI/CD。
- 命令和代码：`pnpm build`、`git commit`。
- 协议和标准：HTTP、JSON、REST。
- 没有通用中文翻译的术语：middleware、monorepo、override。

首次出现核心术语时，优先标注中英对照：

```markdown
Codex 会构建一条指令链（instruction chain）。
```

后续再次出现时保持同一个写法，不要混用多个翻译。

## 句子

- 一句话只说一件事。超过 5 行的段落优先拆分。
- 把英文长定语改成中文短句。
- 避免「被用来」「你需要去」这类生硬表达。
- 技术判断要写清原因、边界和影响。

## 结构

- 技术说明优先使用「先看结论 → 核心机制 → 示例 → 误区 → 总结」。
- 对比关系优先用表格，但表格内不要塞长段落。
- 操作步骤优先用有序列表。
- 代码块必须标注语言；目录树、纯文本结构使用 `txt`。

## 发布前检查

- 中英文之间是否留空格。
- 中文与数字之间是否留空格。
- 中文语境是否使用全角标点。
- 术语是否前后一致。
- 首次出现的核心英文术语是否有解释。
- 段落是否过长。
- 代码块语言是否合理。
