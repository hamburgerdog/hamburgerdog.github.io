# Writing Guide

## 适用范围

这份指南用于新增、编辑、润色 `src/content/blog/*.md` 文章。它总结了当前博客的写作习惯，并补充 agent 写作时需要稳定遵守的规则。

中文技术文章还必须结合 `chinese-documentation.md` 检查中英混排、标点、术语一致性和段落结构。

## Frontmatter

文章必须以 YAML frontmatter 开头：

```yaml
---
title: 文章标题
subtitle: 简短副标题
date: 2026-05-06 15:00:00 +0800
tags: 前端 源码 精选
remark: '面向搜索、分享和列表摘要的一句话说明'
---
```

字段规则：

- `title`：必填，文章展示标题。中文标题可直接使用中文。
- `date`：必填，字符串格式，优先使用 `YYYY-MM-DD HH:mm:ss +0800`。
- `tags`：必填，使用空格分隔；不要写成数组。只能从 `前端`、`源码`、`生活`、`后端`、`工程基建`、`Ai算法`、`性能优化` 中选择，可组合多个标签。`精选` 只在明确需要进入精选区时附加。
- `subtitle`：可选，用于列表页和 SEO 描述，保持短句。
- `remark`：可选，但建议技术文章都写。它应该概括文章覆盖的核心概念、适用场景和相关联文章。

## 标题结构

推荐结构：

1. frontmatter 后写一个与 `title` 一致或高度一致的 `# 一级标题`。
2. 开篇使用引用块或短段落说明文章背景、核心问题和读者能得到什么。
3. 用 `##` 组织主章节，用 `###` 组织细节。文章详情页目录会展示深度小于 4 的标题。
4. 技术文章优先采用「问题背景 → 核心概念 → 过程拆解 → 示例代码/图示 → 总结」。
5. 思考类文章优先采用「观点 → 案例 → 分析维度 → 一句话总结/反问」。

避免：

- 连续出现两个重复的 `#` 标题。
- 标题只写「一、二、三」而没有信息量。
- 章节粒度过碎，导致目录无法快速定位重点。

## 语气与表达

整体语气是中文技术博客风格：直接、解释型、带个人判断，但不做营销化包装。

写法偏好：

- 先讲结论，再补过程。
- 用具体场景打开抽象概念，例如「点击按钮后状态变化，React 需要做什么」。
- 允许使用第一人称，但要服务于经验说明，不写流水账。
- 对复杂机制使用「设想一个场景」「核心思想」「为什么需要」等引导语。
- 在表达判断时给出原因、边界和权衡。
- 结尾可以用「一句话总结」收束，也可以留下一个面向读者的思考问题。

避免：

- 空泛夸张，如「极致」「颠覆」「全网最全」。
- 没有上下文的术语堆叠。
- 长段落连续铺陈。超过 5 行的段落优先拆分。
- 只有结论没有例子，或只有代码没有解释。

## 中文排版

- 中文语境使用全角标点。
- 中英文之间加空格：`React 的 diff 算法`。
- 中文与数字之间加空格：`减少 50% 重复代码`。
- 技术名词保持标准写法：JavaScript、TypeScript、React、Node.js、Vite、HTTP、WebGL、CI/CD。
- 行内代码、命令、字段名使用反引号：`tags`、`pnpm build`。
- 引用外部文章或工具时使用 Markdown 超链接，不直接裸贴长 URL。
- 粗体只用于重点概念或列表项开头，不要整段加粗。
- 表格适合对比概念、生命周期、优缺点；不要用表格承载长段落。

## 技术文章写法

技术文章应降低信息差：

- 首次出现的核心术语给出中文解释或英文全称，例如 MCP（Model Context Protocol，模型上下文协议）。
- 把系统拆成稳定对象、数据流、生命周期或阶段来讲。
- 讲源码或机制时，先写「它解决什么问题」，再写「它如何实现」。
- 代码块前说明读者应该关注什么，代码块后解释结果。
- 涉及性能、成功率、成本下降等数字时，说明口径或前提，避免孤立数字。
- 相关主题之间可以互相引用，例如 React 并发、调度、协调、渲染阶段系列文章。

常用章节模板：

```markdown
## 什么是 XXX？

## 为什么需要 XXX？

## 核心过程

## 示例

## 常见误区

## 一句话总结
```

## 思考类文章写法

思考类文章可以更有作者视角，但仍然保持结构化：

- 开头先给出中心判断，不绕圈。
- 用一个具体案例承载观点，例如简历、项目经历、工具选择。
- 分析维度要明确，如技术深度、问题解决能力、架构视野、业务价值。
- 观点要能被反驳，因此需要给出适用边界。
- 结尾用短句收束，不强行升华。

## 图片和素材

- 新增素材放在 `src/assets/<topic>/`，目录名优先使用文章主题英文或稳定缩写。
- Markdown 引用站内图片时使用相对路径，例如 `![说明](../../assets/write/image0.webp)`。
- 图片 alt 文本写清楚图的作用，不要只写 `image`。
- 优先使用 WebP。已有 PNG/JPG 可通过 `pnpm image2webp` 转换并更新引用。
- 配图要紧贴上下文，避免只有图没有解释。

## 发布前检查

- frontmatter 是否包含 `title`、`date`、`tags`。
- `tags` 是否为空格分隔字符串。
- 标题层级是否从 `#`、`##`、`###` 自然递进。
- 中文与英文、数字之间是否留空格。
- 代码块是否有语言标识，如 `ts`、`tsx`、`bash`。
- 图片路径是否有效，alt 文本是否有意义。
- `remark` 是否能独立说明文章价值。
- 运行 `pnpm build`，确认内容集合和静态构建通过。
